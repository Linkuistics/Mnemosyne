## Config File Location

```
~/.mnemosyne/config.yml
```

The config file is read at the start of every CLI invocation that requires context detection or project analysis (`query --context`, for example). If the file does not exist, built-in defaults are used. Missing top-level sections also fall back to built-in defaults.

---

## Top-Level Structure

```yaml
language_profiles:
  <language-name>:
    markers: [...]
    extensions: [...]
    dependency_file: <string>       # optional
    dependency_parser: <string>     # optional

context_mappings:
  <parser-name>_dependencies:
    <package-name>: [tag, tag, ...]
```

Both sections are optional; if absent, defaults are used. If present, they replace the defaults for that section entirely — there is no deep merge. To extend rather than replace, copy the default content and add your additions.

## Language Profiles

Language profiles configure how Mnemosyne detects that a project uses a particular programming language.

### Profile Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `markers` | list of strings | No | Filenames (or glob patterns with `*`) in the project root that indicate this language is in use. |
| `extensions` | list of strings | No | File extensions (including leading `.`) to scan for in the project root. If any files with this extension exist, the language is detected. |
| `dependency_file` | string | No | Name of the file in the project root that lists dependencies (e.g., `Cargo.toml`, `pyproject.toml`). If present, this file is parsed for dependency names. |
| `dependency_parser` | string | No | Parser to use for the dependency file. Currently supported: `cargo` (parses `[dependencies]` sections), `pyproject` (parses `dependencies = [...]` arrays). |

Language detection is triggered when either:
- At least one marker file exists in the project root (glob patterns are matched against filenames only, not recursively)
- At least one file with a matching extension exists in the project root (non-recursive)

Both `markers` and `extensions` may be empty. A profile with only extensions set detects the language when source files are present without any build system marker.

### Example Profile

```yaml
language_profiles:
  elm:
    markers: ["elm.json"]
    extensions: [".elm"]
    dependency_file: "elm.json"
    dependency_parser: null
```

## Context Mappings

Context mappings translate dependency names (detected from dependency files) into knowledge tags. This is the bridge between "this project uses tokio" and "retrieve global knowledge tagged async, tokio, concurrency".

### Structure

Context mappings are organised by parser name. The key is `<dependency_parser>_dependencies` for each profile that has a `dependency_parser`:

```yaml
context_mappings:
  cargo_dependencies:
    tokio: [async, tokio, concurrency]
    sqlx: [database, sql, async]
    axum: [web, http, api]
```

### How Mappings Are Applied

During `mnemosyne query --context`:

1. The CLI detects languages and dependencies from the project directory
2. For each detected dependency, it looks up `<ecosystem>_dependencies` in `context_mappings`
3. All tags from matching mappings are collected into a tag set
4. Language names are also added as tags directly (e.g., detecting `rust` adds the tag `rust`)
5. The combined tag set is used to query the knowledge store

If a dependency is not in the mappings, it is ignored — it does not generate any tags. This is intentional: only dependencies you've explicitly mapped are significant.

## Default Language Profiles

The built-in defaults include twelve language profiles. These are written to `~/.mnemosyne/config.yml` when you run `mnemosyne init`.

| Language | Markers | Extensions | Dependency Parser |
|----------|---------|------------|-------------------|
| `rust` | `Cargo.toml` | `.rs` | `cargo` |
| `python` | `pyproject.toml`, `setup.py`, `requirements.txt` | `.py` | `pyproject` |
| `haskell` | `*.cabal`, `stack.yaml`, `cabal.project` | `.hs` | `cabal` |
| `ocaml` | `dune-project`, `*.opam` | `.ml`, `.mli` | `opam` |
| `prolog` | `pack.pl` | `.pl`, `.pro` | — |
| `mercury` | `Mercury.options` | `.m`, `.mh` | — |
| `scheme` | — | `.scm`, `.ss`, `.sld` | — |
| `racket` | `info.rkt` | `.rkt` | — |
| `common-lisp` | `*.asd` | `.lisp`, `.cl`, `.lsp` | — |
| `smalltalk` | `.smalltalk.ston` | `.st` | — |
| `idris` | `*.ipkg` | `.idr` | — |
| `swift` | `Package.swift` | `.swift` | — |

Languages without a dependency parser (`—`) have no dependency-to-tag mapping capability; only their language tag is added to context queries.

## Default Context Mappings

The built-in defaults include one context mapping section for Cargo dependencies:

```yaml
context_mappings:
  cargo_dependencies:
    tokio: [async, tokio, concurrency]
    sqlx: [database, sql, async]
    axum: [web, http, api]
    serde: [serialization, serde]
```

This is intentionally minimal. Most Rust projects use many more crates than these four; the expectation is that you extend the mappings based on the libraries you actually work with.

## Adding Custom Languages

To add a language not included in the defaults, add a new entry under `language_profiles`:

```yaml
language_profiles:
  # ... existing profiles ...

  kotlin:
    markers: ["build.gradle.kts", "build.gradle"]
    extensions: [".kt", ".kts"]
    dependency_file: "build.gradle.kts"
    dependency_parser: null

  gleam:
    markers: ["gleam.toml"]
    extensions: [".gleam"]
    dependency_file: "gleam.toml"
    dependency_parser: null

  nix:
    markers: ["flake.nix", "default.nix"]
    extensions: [".nix"]
    dependency_file: null
    dependency_parser: null
```

After adding a profile, the language will be detected in projects that match its markers or extensions, and its name will be added as a tag in context queries.

### Adding a Dependency Parser

If you want dependency-to-tag mapping for a new language, you currently need to implement the parser in the CLI source (see `src/context/detect.rs`). The `dependency_parser` field in the config accepts a string that the detection code maps to a specific parser function.

To add a parser without modifying Rust code, the recommended approach is to use the `null` parser and instead populate `context_mappings` manually with packages you know you use.

## Adding Custom Context Mappings

Extend context mappings to cover additional dependencies in any supported ecosystem:

```yaml
context_mappings:
  cargo_dependencies:
    # Default entries
    tokio: [async, tokio, concurrency]
    sqlx: [database, sql, async]
    axum: [web, http, api]
    serde: [serialization, serde]
    # Your additions
    tonic: [grpc, rpc, protobuf, networking]
    reqwest: [http, web, networking, async]
    tracing: [observability, logging, tracing]
    clap: [cli, argument-parsing]
    anyhow: [error-handling]
    thiserror: [error-handling, type-safety]
    diesel: [database, sql, orm]
    redis: [database, cache, redis]
    rayon: [parallelism, concurrency]

  pyproject_dependencies:
    fastapi: [web, http, api, async]
    sqlalchemy: [database, sql, orm]
    pydantic: [validation, serialization, type-safety]
    pytest: [testing]
    httpx: [http, web, async]
    celery: [async, queuing, background-jobs]
    django: [web, http, orm, database]
```

### Designing Good Tag Mappings

Effective tag mappings connect a dependency to the *topics it embodies*, not just the dependency name. Consider:

- **What problem does this library solve?** `tokio` solves async I/O → tags `async`, `concurrency`
- **What does code using this library need to know?** `sqlx` requires understanding of database connections, async I/O, and SQL → tags `database`, `sql`, `async`
- **What knowledge entries would be relevant when using this library?** An entry about connection pooling is relevant for any `database` tag; an entry about async cancellation is relevant for any `async` tag

Avoid mapping every dependency to dozens of tags — this dilutes signal. Focus on the tags that represent meaningful conceptual areas where you have or expect to have knowledge entries.

### Multiple Ecosystems

If you work in multiple languages with dependency parsers, add a mapping section for each:

```yaml
context_mappings:
  cargo_dependencies:
    tokio: [async, tokio, concurrency]
    # ...

  pyproject_dependencies:
    fastapi: [web, http, api, async]
    # ...
```

Each section corresponds to the `<dependency_parser>_dependencies` key for the language profile that uses that parser.
